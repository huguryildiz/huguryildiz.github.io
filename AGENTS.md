# Repository Guide for Coding Agents

## Mission

This repository is the source for Hüseyin Uğur Yıldız's academic website,
<https://huguryildiz.com>. It is a public scholarly record, not a generic theme demo.
Treat factual accuracy, professional presentation, accessibility, and stable deployment as
hard requirements.

When working here:

1. Inspect the current file and its rendering path before editing.
2. Make the smallest coherent change that satisfies the request.
3. Preserve publication metadata, names, dates, URLs, and research claims unless the user
   supplies evidence for changing them.
4. Validate both the generated site and the affected interaction or layout.
5. Report what was verified and what could not be verified.

Do not invent publications, metrics, affiliations, awards, student records, software
capabilities, citations, or links. Do not silently turn uncertain information into a factual
claim.

## Technology and Deployment

- Static site generator: Jekyll
- Hosting: GitHub Pages
- Production branch: `master`
- Production URL: `https://huguryildiz.com`
- Custom domain declaration: `CNAME`
- Theme dependency: `mmistakes/minimal-mistakes@4.24.0`
- Production workflow: `.github/workflows/jekyll.yml`
- Scheduled metrics workflow: `.github/workflows/update-openalex.yml`

A push to `master` triggers the Pages build and deployment. Do not push, deploy, alter
repository secrets, or manually run a remote workflow unless the user explicitly asks.

## Current Rendering Architecture

The repository contains two presentation paths. Determine which one a page uses from its
front matter before changing layouts, styles, scripts, or navigation.

### Primary academic site

The main pages use `layout: academic`:

- `index.md`
- `_pages/cv.md`
- `_pages/publications.md`
- `_pages/research.md`
- `_pages/service.md`
- `_pages/students.md`
- `_pages/teaching.md`

Their rendering chain is:

```text
page -> _layouts/academic.html -> _layouts/compress.html
                              -> assets/css/redesign.css
```

`_layouts/academic.html` owns the primary navigation, footer, theme toggle, mobile-menu
behavior, analytics tags, and shared page shell. The navigation in this layout is hardcoded;
editing `_data/navigation.yml` does not change these pages.

### Legacy Minimal Mistakes path

Pages without `layout: academic`, including `_pages/404.md`, `_pages/search.md`, and
`_pages/terms.md`, use the local Minimal Mistakes overrides. Their relevant files include:

- `_layouts/default.html` and `_layouts/single.html`
- `_includes/masthead.html`
- `_includes/head/custom.html`
- `_includes/scripts.html`
- `_data/navigation.yml`
- `assets/css/main.scss`, `assets/css/custom.css`, and `_sass/`

Do not assume that a change to one path affects the other. Avoid broad attempts to merge the
two systems unless the task explicitly requires an architectural migration.

Some retained files are not on the active path. In particular,
`_includes/research-metrics.html`, `_includes/publications-scripts.html`, and
`assets/js/pub-chart.js` are not currently referenced by the primary academic pages. Verify
references with `rg` before editing or deleting apparently related files.

The CV conversion scripts are maintenance utilities, not an automatic synchronization
pipeline. `_data/cv.json` is not currently tracked or rendered. Run
`scripts/update_cv_json.sh` only when JSON Resume output is explicitly requested, and review
the generated data rather than assuming that the parser preserved the hand-authored CV.

## Content Rules

- Keep public-facing prose in clear, formal English unless the requested page deliberately
  uses another language.
- Preserve Turkish characters in the displayed name: `Hüseyin Uğur Yıldız`. Metadata may
  intentionally use the ASCII form `Huseyin Ugur Yildiz`; do not normalize globally.
- Treat `_pages/cv.md`, `_pages/publications.md`, and the downloadable CV PDF as independent
  artifacts. A change to one does not automatically update the others.
- When adding a publication, verify authors, title, venue, year, DOI, classification, and the
  local PDF path. Update totals, summaries, and year groupings that are encoded on the page.
- Do not infer bibliometric values. `_data/research_metrics.json` is an OpenAlex snapshot and
  should normally be updated through the metrics script/workflow.
- Use root-relative internal URLs consistently with the current primary pages. For Liquid
  templates and shared layouts, prefer `relative_url` where the surrounding code does so.
- Keep front matter valid and preserve canonical `permalink` values unless a redirect or URL
  migration is part of the task.

## Design and Front-End Rules

`PRODUCT.md` is the product/design contract for the interactive UWSN hero. Its operational
meaning, semantic color assignments, data-honesty requirements, and accessibility constraints
take precedence over decorative changes.

For all UI work:

- Support light and dark themes.
- Check desktop and narrow mobile widths.
- Preserve keyboard operation, visible focus, semantic HTML, useful alternative text, and
  ARIA state updates.
- Respect `prefers-reduced-motion`; do not introduce flashing or essential motion.
- Never encode status or meaning by color alone.
- Avoid fake precision and unlabeled simulated data in research visualizations.
- Prefer existing design tokens, classes, and SVG symbols before adding one-off styles or
  duplicate assets.
- Keep scripts dependency-light and compatible with a static GitHub Pages site.
- Do not add a frontend build system for a small CSS or JavaScript change.

External scripts, fonts, analytics, and embeds have privacy, reliability, and performance
costs. Do not add or replace third-party dependencies without a clear requirement. Preserve
`rel="noopener"` on links that open a new tab.

## Local Development

### Docker path

Use Docker when the host Ruby toolchain is missing or incompatible:

```bash
docker compose up --build
```

This serves the repository at <http://localhost:4000> and combines `_config.yml` with
`_config_docker.yml`.

### Native Ruby path

The GitHub Pages workflow builds with Ruby 3.1. With a compatible local Ruby and Bundler:

```bash
bundle install
bundle exec jekyll serve
```

For a production-style build:

```bash
JEKYLL_ENV=production bundle exec jekyll build
```

Do not modify `Gemfile.lock` merely to work around an outdated system Ruby. Prefer a compatible
Ruby environment or Docker, and keep dependency changes intentional.

### Maintenance scripts

Refresh OpenAlex metrics locally only with an explicit author ID:

```bash
OPENALEX_AUTHOR_ID=A5085505896 python3 scripts/fetch_metrics.py
```

`OPENALEX_MAILTO` is optional. The script requires the Python `requests` package and writes
`_data/research_metrics.json`. It performs a live network request, so inspect the diff after it
runs.

The CV converter additionally requires PyYAML. Its wrapper is interactive:

```bash
scripts/update_cv_json.sh
```

## Validation

There is no automated test suite. Validation must be proportional to the change, but a content
or UI task is not complete solely because the source looks plausible.

Minimum checks:

1. Review `git diff --check` and the focused diff.
2. Run a Jekyll build using Docker or a compatible native Ruby environment.
3. Inspect the affected page at `http://localhost:4000`.
4. Exercise changed controls with keyboard and pointer input.
5. Check both color themes and at least one narrow viewport for layout changes.
6. Inspect the browser console for JavaScript errors when scripts changed.
7. Confirm that changed internal links and local assets resolve.

For shared layout, CSS, navigation, or theme changes, sample all primary academic pages and at
least one legacy page. For publication changes, test type/year/quartile filtering, reset
behavior, counts, DOI links, and PDF links. For hero or research-map changes, verify reduced
motion, keyboard behavior, responsive layout, labels, and semantic color consistency.

If the environment prevents a build or browser check, do not claim success. State the exact
missing dependency or failed command and report the checks that did run.

## Change Discipline

- Start with `git status --short`; preserve unrelated user changes.
- Use `rg`/`rg --files` to locate active references before editing.
- Do not edit generated `_site/` output or dependency directories.
- Do not edit remote or installed theme files. Override behavior in tracked local layouts,
  includes, Sass, CSS, or JavaScript.
- Do not remove retained legacy files simply because the primary pages do not reference them.
- Avoid unrelated formatting, mass rewrites, and broad renames.
- Never commit secrets, tokens, private student information, drafts, or local machine artifacts.
- Do not commit, push, tag, or deploy unless the user requests it.

When handing off, state the files changed, the user-visible effect, the validation performed,
and any unresolved uncertainty. Keep the report factual and concise.
